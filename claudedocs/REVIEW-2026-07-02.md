## 1. Executive Verdict

**Overall quality: high.** neo-rs is a mature, professionally-structured port. Codec/IO layers (`neo-io`, `neo-serialization`), crypto (`neo-crypto`), config, and the execution engine show genuine byte-for-byte discipline against the C# v3.10.0 reference, with extensive parity tests and careful hardfork gating. Architecture is clean and reth-style, with a coherent composition root.

**Is "100% correct and complete / byte-for-byte v3.10.0 parity" justified? No — not yet.** The claim is aspirational, and the audit surfaces concrete, verified divergences that break it. Crucially, several are **consensus- or handshake-relevant**, not cosmetic:

- **Live consensus faults exist today.** A dBFT backup can commit-sign a proposal whose transactions it does not hold (`neo-consensus`), and the consensus driver reuses a frozen startup snapshot so validator-set/NextConsensus reads go stale across committee-refresh heights (`neo-node`). Both are verified `high` and both diverge from the C# reference on real networks.
- **P2P handshake is wire-incompatible.** ArchivalNode capability serializes an extra `0x00` byte that C# does not emit, and it is advertised on **every** default Version handshake — desyncing the capabilities array in both directions with real C# peers.
- **VM-reachable execution divergence.** `StdLib.jsonSerialize` succeeds on invalid UTF-8 where C# FAULTs, and `CryptoLib.recoverSecp256K1` accepts 64-byte EIP-2098 signatures where C# returns null (Echidna-gated, **live on mainnet**) — either lets the same transaction produce different execution results on C# vs. Rust nodes.
- **Latent state-root split.** The `HF_Gorgon` `GetCandidateVote` override is missing from `NeoToken.post_persist`; dormant today (Gorgon unscheduled), but a hard fork the instant Gorgon activates.
- **Large completeness gaps in the parity surface.** The signed StateRoot subsystem (witness/vote/StateValidators multisig) is unimplemented, and a broad set of `unverified` native-contract items (committee `EffectiveVoterTurnout` fallback, `Policy.IsBlocked` candidate filtering, conflict-stub persistence, deploy/update validation, several StdLib edge cases) — if confirmed — are consensus-critical.

> **Follow-up note:** every summary-level item originally flagged "critical but unverified" was independently re-verified. Four were **refuted as false positives** (VM gas metering, NeoToken governance fallbacks, RocksDB snapshot isolation, native fee-overflow) and one was **downgraded** (static-files checksum → local-integrity only). The confirmed findings below survived that second adversarial pass.

**Biggest risks to consensus parity, ranked:**
1. **dBFT commit-without-transactions** and **stale consensus snapshot** — active, exploitable divergence in the running consensus path.
2. ~~Unverified NeoToken governance gaps~~ — **follow-up verified: REFUTED.** The `EffectiveVoterTurnout` standby fallback (`neo_token/storage/mod.rs:526-556`, integer-safe `voters*5 >= total` form) and `Policy.IsBlocked` candidate filtering (`storage/mod.rs:600,650-658,835`) are both correctly implemented; the cross-cutting reviewer missed the logic in `storage/`.
3. **jsonSerialize UTF-8** — VM-reachable state divergence.
4. **ArchivalNode handshake byte** — breaks interop with C# peers on default config.
5. **Dormant HF_Gorgon divergences** (NeoToken reward, CryptoLib strict variants) — time-bombs at hardfork activation.

The honest framing: **the deterministic data-format layers are close to parity; the live consensus/networking coordination layers and the governance/state-root completeness surface are where parity currently breaks.**

## 2. CRITICAL and HIGH Findings

### 2a. Protocol-correctness / Consensus

**[HIGH] dBFT backup commits without holding proposal transactions**
`neo-consensus/src/service/handlers/prepare.rs:269` — `check_prepare_responses()` broadcasts a Commit once `M` prepare responses are counted, but omits the `has_missing_proposed_transactions()` gate that the sibling `check_commits()` (`handlers/commit.rs:172`) correctly has. C# `CheckPreparations` requires **both** `PreparationPayloads >= M` **and** `TransactionHashes.All(...ContainsKey)`. **Impact:** a backup with `prepare_request_received=true` reaching `M` from other validators' PrepareResponses signs a Commit toward finalization for a block whose transactions it never received/validated — weakens dBFT's validation guarantee, diverges from reference. Reachable by a single sender. **Fix:** add `if self.context.has_missing_proposed_transactions() { return Ok(()); }` at the top of `check_prepare_responses()`, mirroring `commit.rs:172`.

**[HIGH] Consensus driver reuses a frozen startup DataCache for every round**
`neo-node/src/consensus/mod.rs:315` (threaded into `configure_round` at :318/:371, ledger guards at :354/:357, proposal cache at :448). The snapshot from `node/mod.rs:774-775` is a memoizing `DataCache` (`track_reads_in_write_cache=true`, overlay shared via `Arc`); nothing rebuilds it per round. Committee/validator reads (`next_block_validators`, `next_consensus_address_for_block` → `read_committee_with_votes` → `snapshot.get(committee_key())`) return the **stale round-1 committee** after a committee-refresh height. **Impact:** wrong validator set/index, wrong NextConsensus (bft_address of wrong set), wrong multisig ordering → fork from correctly-updated nodes / unverifiable witness. C# `ConsensusContext.Reset` takes a fresh snapshot each round. Conditional on a driver process spanning a committee boundary (every `committee_members_count` blocks, e.g. 21 on mainnet). **Fix:** take a fresh store snapshot/`DataCache` at the start of each round (start and each `Imported`) for `configure_round`, ledger guards, and proposal verification.

**[HIGH] jsonSerialize / JSON ByteString: silent UTF-8 corruption vs. C# FAULT**
`neo-serialization/src/codec/json_serializer.rs:79` (and `:83`). ByteString/Buffer values use `from_utf8_lossy` (emits U+FFFD), while map **keys** (`:149`) correctly hard-error on invalid UTF-8. C# `JsonSerializer` uses `StrictUTF8` (`throwOnInvalidBytes=true`) for values → VM FAULT. Reachable via `StdLib.jsonSerialize` (`neo-native-contracts/src/std_lib/serialization.rs`), an interop any contract can invoke. **Impact:** same transaction FAULTs on C# but succeeds here → divergent execution results / fork risk. **Fix:** change lines 79 and 83 to `String::from_utf8(bytes).map_err(|_| CoreError::other("Invalid UTF-8 in byte string"))?`, mirroring line 150.
> Note: cross-cutting item `jsonSerialize encodes ByteString as base64` (`std_lib/serialization.rs:26`, unverified) targets the same interop and should be resolved together — confirm the value encoding is strict-UTF-8, not base64 or lossy.

**[HIGH — verified] CryptoLib.recoverSecp256K1 accepts 64-byte EIP-2098 signatures that C# rejects (live-mainnet consensus fork)**
`neo-crypto/src/keys/signature.rs:96` accepts both 65- and 64-byte secp256k1 signatures (`if signature.len() != 65 && signature.len() != 64`), with EIP-2098 y-parity folding at `:119-124`. The native `recoverSecp256K1` handler (`neo-native-contracts/src/crypto_lib/mod.rs:321-323`) passes the raw, attacker-controlled signature straight through with no length gate; `Some(key)` pushes the compressed pubkey, `None` pushes `StackItem::Null` (`:324-330`). C# `Crypto.ECRecover` requires **exactly 65 bytes** and throws → `RecoverSecp256K1` returns `null`. **Impact:** for a 64-byte compact signature that recovers successfully, neo-rs pushes a valid `ByteString` where C# pushes `Null`; any contract branching on the result (null-check, or using the key as a script-hash/authorization) diverges → different state root → chain split. The method is `HF_Echidna`-gated (`crypto_lib/metadata.rs:152`) and Echidna is **live on MainNet since block 7,300,000**, so this is reachable today by any deployed contract that calls it with a 64-byte arg. The in-repo test never asserts 64-byte input returns null. **Fix:** reject any signature whose length is not exactly 65 in the `recoverSecp256K1` consensus path; keep the 64-byte/EIP-2098 form only behind a non-consensus API. Add a parity test pinning 64-byte input → null. (Belt-and-suspenders: diff against neo-project/neo `Crypto.cs` at the v3.10.0 tag to confirm the `!= 65` throw.)

**[HIGH] ArchivalNode capability serializes an extra `0x00` byte (handshake wire divergence)**
`neo-payloads/src/p2p_payloads/handshake/node_capability.rs:195` (size `:171`, deserialize `:236`). Rust emits type `0x11` + a literal `0x00` and requires a trailing zero on read. C# `ArchivalNodeCapability` is a payload-less marker (empty Serialize/Deserialize, Size==1). Capabilities are a var-int count + directly concatenated items with **no per-item framing**, so the stray byte is unframed and corrupts the stream. Advertised unconditionally on every default handshake (`neo-network local_identity.rs:114-127`). **Impact:** Rust→C# leaves a stray `0x00` C# mis-parses as a spurious `UnknownCapability`; C#→Rust `ensure_zero_byte` consumes the next capability's type byte or hits EOF. Both interop directions break. **Fix:** make ArchivalNode a zero-payload marker (emit only `0x11`, read nothing, size 1); add round-trip tests pinned to captured C# bytes for every capability type.
> Caveat: no in-repo golden vector pins C#'s bytes; confirmation rests on knowledge of C# source. Capture a real C# handshake to lock this.

**[HIGH] NeoToken.post_persist omits HF_Gorgon `GetCandidateVote` override (latent state-root split)**
`neo-native-contracts/src/neo_token/mod.rs:533` (loop `:526-547`). The voter-reward loop divides by cached committee votes with no HF_Gorgon branch; C# replaces `votes` with live `GetCandidateVote` under `HF_Gorgon`. Cached votes (blocked/standby→0) differ structurally from `GetCandidateVote` (registered→real, unregistered→-1), so the divisor and `votes>0` guard differ → different `Prefix_VoterRewardPerCommittee` → state-root divergence. **Dormant:** `HfGorgon=6` is defined but unscheduled (tests assert disabled even at `u32::MAX`). **Fix:** gate the per-member vote on HF_Gorgon activation; re-read live candidate vote when enabled. Add a regression test pinning accrued reward bytes under an enabled-Gorgon setting.

**[MEDIUM] Oracle redirect handling ignores HTTP status code**
`neo-oracle-service/src/https/process.rs:98`. Client disables auto-redirect; code checks the `LOCATION` header (`:97-131`) before the status code (`:133-142`), so any non-3xx response carrying `Location` (e.g. 200/201) is treated as a redirect — body never returned. Not SSRF (targets re-validated), but diverges from C# which gates redirect on 3xx → different `OracleResponseCode`/data submitted to consensus in a mixed C#/Rust network. **Fix:** wrap the LOCATION block in `if response.status().is_redirection() { ... }`.

**[MEDIUM] Oracle per-chunk 8KB cap can falsely reject legitimate responses**
`neo-oracle-service/src/https/process.rs:219`. Rejects any single stream chunk > `MAX_CHUNK_SIZE` (8KB) with `ResponseTooLarge`, independent of the 64KB cumulative limit. `reqwest::bytes_stream()` chunk sizes reflect server/TCP framing, not 8KB. A legitimate <64KB response delivered in one large frame is wrongly rejected; C# has only a total-size limit → an oracle node reports `ResponseTooLarge` where C# reports `Success`. **Fix:** remove/downgrade the per-chunk check; rely on the cumulative (`:229`) and final (`:243`) limits.

### 2b. Bugs / Security

**[HIGH] Negative `effective_pooled_fee` allows an insufficient-GAS admission bypass**
`neo-mempool/src/pool/memory_pool.rs:549` (and `:666`). `effective_pooled_fee = &pooled_sender_fee - &rebate` is unbounded; passed into `verify_transaction` → `admission/verification.rs:243` where `expected_fee = system_fee + network_fee + pooled_sender_fee` is under-charged when negative. **Root cause** (mis-located in original finding): `check_conflicts` (`memory_pool.rs:100-147`) builds `Vec<PoolItem>` without dedup across its two steps, so a mutual conflict double-counts a tx's fee in `conflict_rebate` (`:212`) while `pooled_sender_fee` counts it once. Attacker-reachable: same sender submits A then B; the admission gate at `:144` also double-counts, requiring only `2*A.network_fee < B.network_fee`. **Scope:** mempool-local (rebate cleared on persist `:435`; not on-chain/consensus), so the underfunded tx still fails real GAS deduction at execution — **no consensus split**, but mempool integrity corruption + possible relay of an underfunded tx. **Fix:** dedup the conflict list so a tx is counted once (root fix); a `max(0)` clamp on the subtraction masks the symptom but leaves `conflicts_fee_sum` inflated.

**[HIGH] PKCS#11 HSM redeem script uses a WRONG hardcoded CheckSig syscall hash**
`neo-hsm/src/providers/pkcs11.rs:535` (hash at `:546`). `signature_redeem_script` hardcodes the `System.Crypto.CheckSig` hash as `[0xe9,0x27,0xec,0x2b]` with a comment falsely claiming C#-verified. Correct value from `neo_vm_rs::interop_hash` is `0x27b3e756` → LE bytes `[0x56,0xe7,0xb3,0x27]`. The wrong bytes flow into `UInt160::from_script` (`:370`), producing an **incorrect script hash/address** for the HSM validator identity used in `can_sign` and as NextConsensus. **Impact:** a PKCS#11-HSM-backed validator derives an identity that doesn't match its real account — breaks consensus signing and wallet interop. Untested: the unit test (`tests/providers/pkcs11.rs:38-47`) only asserts bytes 0/1/35, never 36..40. **Fix:** use `RedeemScript::signature_redeem_script` / `check_sig_hash()` from neo-vm instead of hardcoding; add an assertion on `script[36..40]`.

### 2c. Completeness (consensus-relevant)

**[HIGH] StateRoot payload lacks Witness field and IVerifiable/IInventory signing surface**
`neo-state-service/src/protocol/state_root.rs:18`. `StateRoot` is `(version, index, root_hash, cached_hash)` with no `witness`, no `get_script_hashes_for_verifying`, no `Verify`/`GetSignData`. A `Vote` struct exists but is wired into nothing. The local unwitnessed persistence path (var-int(0) witness) is correct; the **state-validator consensus subsystem** (vote aggregation, StateValidators multisig, signed P2P StateRoot broadcast/validation, `GetSignData(network)=network||Hash`) is unimplemented. **Impact:** node cannot produce or validate network-signed state roots as C# `StateService` does. **Fix:** implement the signed path, or explicitly document that only local state roots are supported so no caller assumes wire-compatible signing.

**[HIGH] StateRoot verifier accepts on trie-hash match only; no validator-signature check**
`neo-state-service/src/validation/verification.rs:84`. Verifier recomputes the root and accepts iff `recomputed == candidate`; no StateValidators multisig verification, no vote aggregation. **Fail-closed** (stricter than C#, not exploitable) but a real capability gap: fast/light-sync nodes cannot trust validator-signed roots, and this node cannot participate in producing them. Same root cause as the finding above. **Fix:** when signed state roots are implemented, also accept candidates whose StateValidators witness verifies, and wire in Vote collection.

**[REFUTED after follow-up verification] NeoToken governance gaps flagged by cross-cutting analysis**
The two alleged consensus-critical gaps were re-verified against the actual code and are **false positives**:
- `EffectiveVoterTurnout` fallback to StandbyCommittee **is implemented** — `neo_token/storage/mod.rs:526-556` (`compute_committee_members`): turnout branch at :543-547 uses the integer-exact `voters_count * 5 >= NEO_TOTAL_AMOUNT` form of C#'s `< 0.2M` check, and the `candidate_count < committee_count` branch at :549-554 also falls back to standby; `standby_committee_with_registered_votes` (:558-587) matches C#'s standby-with-votes construction. Ordering (votes desc, pubkey asc) matches C# at `storage/candidates.rs:96-120`.
- `Policy.IsBlocked` filtering **is implemented** on every selection path — live-ranking scan skips blocked candidates (`storage/mod.rs:600`, `:650-658`), the standby fallback checks `candidate_is_blocked` (:571), and `getCandidates` filters at :835 using the correct signature-contract account derivation (`candidates.rs:79-90`).

Remaining items to double-check (not refuted, just unexamined): PostPersist voter-reward accumulation on refresh blocks (`neo_token/mod.rs:481`, overlaps the verified HF_Gorgon finding) and `OnBalanceChanging` vote side-effect parity (`transfers.rs:10`).

### 2d. Architecture (cross-cutting, unverified)

**[CRITICAL — unverified] Plugin/RPC upward-dependency layering violations**
`neo-rpc/Cargo.toml:125` and `neo-oracle-service/Cargo.toml:49` depend directly on concrete higher-layer crates (`neo-system`, `neo-blockchain`, `neo-network`, `neo-wallets`, `neo-mempool`, `neo-state-service`), contradicting `docs/architecture.md:16` ("strictly layered, downward only"). Under-abstraction root cause: `neo-runtime` (`lib.rs:38`) lacks `LedgerQueryService`/`MempoolQueryService`/`StateQueryService`/`WalletSigningService` traits, so consumers import concrete types. **These are `critical`/`high` only as architecture-quality/testability issues — no protocol/consensus impact; treat as design debt, not parity blockers.** **Fix:** define query-service traits in `neo-runtime`; have `neo-rpc`/`neo-oracle-service` depend on traits + `RpcServiceContext`/`OracleServiceContext`; wire concretes at composition time. Also update `architecture.md` to stop overstating strictness (`docs/architecture.md:16`).

## 3. MEDIUM Findings (de-duplicated)

### Completeness gaps in native contracts (unverified — verify against C#)
- **getBlock missing `IsTraceableBlock` gate** (`ledger_contract/mod.rs:230`) — helper exists and is used for tx getters but not getBlock; C# returns null outside the traceable window.
- **Conflicts-attribute stubs not persisted in `on_persist`** (`ledger_contract/mod.rs:160`) — `contains_conflict_hash` reads stubs, but new conflicts are never written → conflict-avoidance breaks over time.

## 4. LOW / Professionalism Themes (aggregate)

A consistent set of low-severity, non-consensus nits — all verified real, none affect node behavior:

- **Misleading comments / wrong error text:** quote-escape doc comment (`neo-serialization/src/json/escape.rs:270`); oracle `MIN_GAS_FOR_RESPONSE` message says "0.1 datoshi" instead of "0.1 GAS" (`oracle_contract/mod.rs:319`); PKCS#11 comment drift on the CheckSig hash (compounds the HIGH bug in 2b).
- **Silent-failure / observability gaps:** oracle signature assembly silently returns on key/sign failure (`.../transactions/signature.rs:26` and `:34`); TEE wallet swallows sealed-key load failures with a warning (`neo-tee/.../tee_wallet.rs:85`); Azure HSM misclassifies a sign-time token error as `Init` (cosmetic — all HsmError variants collapse to one ConsensusError) (`neo-hsm/.../azure.rs:103`); node.rs ledger pre-checks swallow errors via `.unwrap_or(false)` inconsistent with adjacent `?` (`neo-system/.../composition/node.rs:352`, plausible/low).
- **Magic numbers / hardcoded bytes:** NEP-2 `0xe0` flags byte unexplained (`neo-wallets/.../key_pair.rs:287`, info); hardcoded storage_price/exec_fee_factor init later overwritten by policy refresh (`neo-execution/.../state.rs:82-83`).
- **Dead / misleading API:** `Witness::verify_signature()` — unused, simple-sig-only, would reject valid witnesses if wired in (`neo-payloads/.../witness.rs:162`); `ConsensusPayload::get_sign_data` — dead, non-protocol signing layout, misleading "hash for signing" doc (real path is `dbft_sign_data`) (`neo-consensus/src/messages/mod.rs:101`).
- **Test-coverage gaps:** HSM redeem-script test skips the security-relevant CheckSig bytes (`neo-hsm/tests/providers/pkcs11.rs:38`); BigDecimal `Mul` unchecked u8 decimals overflow, unreachable via protocol (`neo-primitives/.../big_decimal.rs:334`); Signer AllowedGroups reject the 0x00 point-at-infinity encoding C# accepts, only affects effectively-invalid txs (`neo-payloads/src/witness_rule/helpers.rs:38`).
- **neo-crypto hygiene (verified, non-consensus):** dead/unverified BLS12-381 *signature scheme* `Bls12381Crypto` and a curve-guessing `verify_signature_bytes` with no callers — audit surface with no benefit, remove or gate (`neo-crypto/src/curves/bls12381.rs:26`, `keys/signature.rs:518`); BIP-32 child-key derivation uses variable-time `num_bigint` arithmetic on secret keys instead of the curve's constant-time `Scalar` (`keys/bip32.rs:82`, wallet-side timing hygiene); `ECPoint` derives `serde::Deserialize` bypassing on-curve validation while `Ord::cmp` does `.expect()` re-parse — latent panic if an `ECPoint` is ever deserialized from untrusted input into a `BTreeMap`/sorted validator vec (`curves/ecc.rs:100,521`); `ECPoint` docs claim secret-material zeroization though it only holds public keys (misleading) (`curves/ecc.rs:106`).

**Recommended handling:** batch-fix the dead/misleading-API items (`Witness::verify_signature`, `ConsensusPayload::get_sign_data`) since they are foot-guns for future callers; fold the rest into normal cleanup.
