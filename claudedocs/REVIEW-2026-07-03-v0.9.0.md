## 1. Verdict

**neo-rs v0.9.0 is fundamentally sound and close to parity, but not yet "100% correct & complete."** The deterministic layers are excellent; the gaps are concentrated in **dBFT view-change**, the **signed StateRoot subsystem**, and a handful of native/wallet/oracle parity details. Two results are especially reassuring:

- **Serialization is byte-exact.** Block, Header, Transaction, Signer, Witness, all 5 TransactionAttribute variants, ExtensiblePayload, the StackItem `BinarySerializer`, and the `GetSignData` preimage were all verified byte-identical to C# v3.10.0 (only 2 low decoder-strictness nits on *invalid* input). No consensus byte divergence.
- **The neo-vm-rs 0.2.0 migration is clean.** Host↔pure StackValue conversions preserve identity, no production code relies on the old structural equality, and serialization skips the id — wire bytes unchanged. No correctness bug introduced by the migration.

**But the re-review found real issues the first pass missed — and one regression the first pass *caused*:**

## 2. Critical / High (each C#-verified)

### ⚠️ Regression I shipped in 0.9.0 — REVERT
**[HIGH] Oracle redirect is now gated on 3xx, diverging from C#.** `neo-oracle-service/src/https/process.rs:98`. My 0.9.0 change wrapped the redirect in `if response.status().is_redirection()`. But C# `OracleHttpsProtocol` (verified) follows a redirect whenever the `Location` header is present, **regardless of status code** (`if (message.Headers.Location is not null) { uri = ...; }`). The *original* Rust code (follow any `Location`) was correct; my fix, based on the first review's incorrect C# claim, introduced the divergence. **Fix: revert to following any response carrying a `Location` header.**

### Consensus — dBFT view change (verified vs neo-modules `DBFTPlugin`)
> Note: the canonical v3.10.0 dBFT plugin is not cleanly tagged (absent from the neo v3.10.0 tree; neo-modules tags stop at v3.7.5). These were verified against neo-modules **master**, which carries the newer `ChangeViewReason` enum the Rust targets. The wire-format change (RejectedHashes) is high-stakes — confirm against your canonical dBFT reference before changing bytes.

- **[CRITICAL] ChangeView can move the view BACKWARD.** `neo-consensus/src/service/handlers/change_view.rs:211` + `context/mod.rs:307`. C# `CheckExpectedView` begins with `if (context.ViewNumber >= viewNumber) return;`. The Rust `change_view()` calls `reset_for_new_view(new_view)` **with no guard**, and ChangeView is exempt from the `payload.view_number == context.view_number` filter — so a stale ChangeView from a lagging node (`new_view = sender_view + 1 ≤ our view`) resets this validator to a **past** view, re-opening a decided round. **Fix: early-return in `change_view` when `new_view ≤ self.context.view_number`.** (Safe to add regardless of reference ambiguity.)
- **[HIGH] ChangeView serializes a non-existent `RejectedHashes` array.** `neo-consensus/src/messages/change_view.rs:85`. For the `TxRejectedByPolicy`/`TxInvalid` reasons the Rust appends a `UInt256[]`; C# `ChangeView.Serialize` writes **only** `Timestamp(8)+Reason(1)` (`Size = base + sizeof(ulong)+sizeof(ChangeViewReason)`). Since the ChangeView body is signed, a policy-rejection ChangeView from a Rust node won't verify on C# peers. The in-code comment claiming C# parity is unsupported. **Fix: serialize only timestamp+reason.**
- **[HIGH] View change never broadcasts the node's own ChangeAgreement.** `handlers/change_view.rs:211`. C# `CheckExpectedView`, on reaching M agreements, broadcasts `MakeChangeView(ChangeViewReason.ChangeAgreement)` before `InitializeConsensus` (if its own last change-view < target). Rust only resets locally, so a lagging validator jumps views silently and peers never observe its agreement — a convergence gap. **Fix: broadcast own ChangeView(ChangeAgreement) when M is first reached.**
- **[MEDIUM/HIGH] PrepareResponse from the primary index is counted toward M.** `service/handlers/prepare.rs:155`. C# ignores a PrepareResponse whose validator index is the primary (the primary's preparation is the PrepareRequest). Counting it can double-count toward `M`.

### Native / execution parity
- **[HIGH] `ContractManagement.destroy` missing the HF_Gorgon block-before-erase branch.** `neo-native-contracts/src/contract_management/mod.rs:345`. Verified against the real v3.10.0 `ContractManagement.cs`: `DestroyV0 [ContractMethod(true, HF_Gorgon)] → DestroyInternal(false)` and `DestroyV1 [ContractMethod(HF_Gorgon)] → DestroyInternal(true)`. Post-Gorgon runs `Policy.BlockAccount` + `CleanWhitelist` **before** erasing; pre-Gorgon erases first. Rust implements only one ordering → state-root divergence at/after Gorgon. **Fix: gate the ordering on HF_Gorgon.**
- **[HIGH] `StdLib.jsonDeserialize` large-integer fork.** `neo-serialization/src/codec/json_serializer.rs:218`. Verified: C# routes every JSON number through `double` (Neo.Json `JNumber.Value` is a `double`, no MAX_SAFE_INTEGER guard) then `BigInteger.Parse(value.ToString())` post-Basilisk; Rust keeps exact `i64/u64`. For integer literals in `(2⁵³, i64/u64 max]` the results differ, and `jsonDeserialize` re-serializes the item → divergent execution results. **Fix: route all numbers through f64 as C# does.**

### Completeness — signed StateRoot subsystem (7 findings)
- **[HIGH] `StateRoot` has no `Witness` field** (`neo-state-service/src/protocol/state_root.rs:19`) and **no signed P2P broadcast/inbound dispatch** (`message_type.rs:10`); **`GetSignData(network)` omits the network magic** (`state_root.rs:65`); the **cache stores unsigned roots only** (`mpt_store.rs:1360`); **`getstateroot` RPC always returns an empty `witnesses` array** (`rpc_server_state/mod.rs:570`); the **verifier recomputes the root but never checks BFT signatures** (`validation/verification.rs:84`, marked dormant in-code). The node cannot produce or validate network-signed state roots as C# `StateService` does — the single biggest completeness gap for a v3.10.0 parity claim.

### Wallet / oracle / node
- **[MEDIUM] NEP-2 hardcodes scrypt N=16384/r=8/p=8**, ignoring NEP-6 wallet-specified params (`neo-wallets/src/crypto/key_pair.rs:234`) → can't decrypt wallets built with custom params (standard wallets unaffected).
- **[MEDIUM] Wallet builds only single-signature witnesses** (`wallet_account.rs:270`); multisig contracts produce invalid witnesses. Plus signer/witness 1:1 alignment loss in `sign_transaction` (`nep6.rs:438`).
- **[MEDIUM] SSRF check inspects only the first resolved DNS address** (`neo-oracle-service/src/https/security.rs:28`); C# checks all — multi-address/rebinding bypass (optional service).
- **[MEDIUM/HIGH] neo-system composition root swallows storage errors on the oracle-response admission path** (`neo-system/src/composition/node.rs:342,355` — `.unwrap_or(false)`), admitting+relaying a possibly-duplicate/conflicting tx on a transient storage error; and duplicates tx pre-verification protocol logic that belongs one layer down.
- **[MEDIUM] Observability error path logs the raw (secret-bearing) endpoint URL** unredacted (`neo-node/src/node/observability.rs:490`).

## 3. Medium — efficiency / duplication / latent

- **Consensus verifies each inbound message signature twice** (recompute hash + ECDSA verify) (`neo-consensus/src/service/lifecycle.rs:125`).
- **Reverified mempool txs are never rebroadcast** (C# `ReverifyTransactions` `RelayDirectly` gap) (`neo-mempool/.../memory_pool.rs:510`).
- **`clone_cache` fast/slow merge paths diverge** for a delete of a store-absent key (spurious `Deleted` marker; benign today but the paths are advertised as equivalent) (`neo-storage/.../data_cache/cache.rs:294`).
- **Duplication:** CheckSig redeem-script bytes hand-rolled in 3 sites (`neo-hsm/.../pkcs11.rs:520`); native-contract IDs re-declared as private consts vs public `::ID` (`native_persist.rs:87`); Policy attribute-fee constants duplicated across `neo-payloads`/`neo-native-contracts`; `apply_delete` NotFound inserts an empty `StorageItem` vs C#'s stored value; `StorageKey::length()` off-by-one vs `to_array()`.
- **NeoFS HTTP-gateway header/hash formats differ from C# protobuf** (`neo-oracle-service/.../neofs/http/requests/header.rs:25`).
- **Parity-critical NeoToken on-persist is instrumented with neo-runtime telemetry stage markers** (layering smell) (`neo_token/mod.rs:25`).
